1. ภาพรวมระบบ

1.1 ข้อมูลทั่วไป

LMDS (Logistics Master Data System) เป็นระบบจัดการข้อมูลหลักและจับคู่ข้อมูลการจัดส่งสินค้า พัฒนาบนแพลตฟอร์ม Google Apps Script ทำงานร่วมกับ Google Sheets เป็นฐานข้อมูล ระบบนี้รับข้อมูลการจัดส่งจาก SCG API ที่มีคุณภาพต่ำ (ชื่อสกปรก ที่อยู่ไม่เป็นมาตรฐาน พิกัดไม่แม่นยำ) ผ่านกระบวนการทำความสะอาด ทำ Normalize จับคู่กับข้อมูลหลัก และสร้าง Transaction สำหรับการติดตามการจัดส่งจริง

1.2 สถาปัตยกรรม 6 ชั้น

ระบบออกแบบตามสถาปัตยกรรมแบบชั้น (Layered Architecture) จำนวน 6 ชั้น แต่ละชั้นมีหน้าที่ชัดเจนและทำงานแบบ One-Way Dependency (ชั้นบนเรียกชั้นล่างเท่านั้น) การแบ่งชั้นแบบนี้ช่วยให้สามารถทดสอบและบำรุงรักษาแต่ละส่วนได้อย่างอิสระ โดยไม่กระทบต่อส่วนอื่นๆ ของระบบ

1.3 โมเดลข้อมูล Trinity Framework

หัวใจของระบบคือ Trinity Framework ซึ่งยึดหลักการว่าการจัดส่งทุกครั้งต้องมี 3 เสาหลัก: WHO (ผู้รับ), WHERE-Address (ที่อยู่), WHERE-Coordinate (พิกัด) การออกแบบนี้ช่วยให้สามารถแยกแยะและจัดการแต่ละมิติของข้อมูลได้อย่างอิสระ โดยมีตาราง M_DESTINATION เป็นจุดตัดที่เชื่อมทั้ง 3 เสาเข้าด้วยกัน พร้อมรหัส master_uuid ที่ใช้ในการเชื่อมโยง Alias ข้ามเอนทิตี้

3. สถาปัตยกรรมระบบเชิงลึก

3.1 Core System (Group 0)

Core System เป็นรากฐานของระบบทั้งหมด ประกอบด้วย 6 ไฟล์ที่ต้องถูกโหลดก่อนไฟล์อื่นๆ ตามลำดับหมายเลข 01_Config.gs เป็น Single Source of Truth สำหรับค่าคงที่ทั้งหมด รวมถึงชื่อชีต 21 ชีต, Index Constants 13 ชุด (PERSON_IDX, PLACE_IDX, FACT_IDX ฯลฯ), AI_CONFIG สำหรับควบคุมการจับคู่, และ SCG_CONFIG สำหรับการเชื่อมต่อ API

กลไก validateConfig(): ทำงานทุกครั้งที่เปิด Spreadsheet (ผ่าน onOpen) ตรวจสอบว่า THRESHOLD_AUTO > THRESHOLD_REVIEW > THRESHOLD_IGNORE และตรวจ SCHEMA.length vs IDX.keys ตรงกันทุกตาราง หากไม่ตรงจะโยน Error ทันทีเพื่อป้องกันการทำงานด้วย Config ที่ผิดพลาด

00_App.gs เป็น Gateway หลัก ควบคุม Custom Menu และ Pipeline Trigger ฟังก์ชันสำคัญคือ runFullPipeline() ที่รัน 3 ขั้นตอนต่อเนื่อง: Load Source, Normalize, Match Engine โดยมี LockService ป้องกันการรันซ้อน, Cache Invalidation ก่อนเริ่ม Pipeline, และ Diagnostic Summary หลังเสร็จสิ้น

3.2 Master Database (Group 1)

กลุ่มนี้รับผิดชอบการจัดการข้อมูลหลักทั้งหมด ตั้งแต่การทำ Normalize ชื่อและที่อยู่, การสร้าง/อัปเดต Master Record, ไปจนถึงการจับคู่ข้อมูลดิบกับ Master ที่มีอยู่ โมดูลสำคัญที่สุดคือ 10_MatchEngine.gs ซึ่งเป็นหัวใจของ Pipeline และเป็น Single Writer เดียวสำหรับตาราง M_ALIAS ตามรูปแบบ Single Writer Pattern

บริการ Normalize (05_NormalizeService.gs)

ทำหน้าที่ทำความสะอาดข้อความทั้งหมดก่อนเข้าสู่กระบวนการจับคู่ รวมถึงการตัดช่องว่าง, แปลงเป็นตัวพิมพ์เล็ก, ลบคำนำหน้า/คำลงท้ายที่ไม่จำเป็น, และ Normalization ภาษาไทยเฉพาะ เช่น การจัดการสระและวรรณยุกต์ ข้อมูลที่ผ่าน Normalize แล้วจะถูกเก็บในคอลัมน์ normalized_name แยกจาก canonical_name (ชื่อดั้งเดิม) เพื่อให้สามารถค้นหาเปรียบเทียบได้โดยไม่สูญเสียข้อมูลต้นฉบับ

บริการ Person & Place (06, 07)

รับผิดชอบการสร้าง, ค้นหา, อัปเดต Master Record ของบุคคลและสถานที่ ทั้งสองโมดูลมีกลไก Cache ภายในเพื่อลดการอ่านชีต และใช้ master_uuid เป็นตัวเชื่อมโยงกับระบบ Alias สากล ฟังก์ชันสำคัญได้แก่ resolvePerson() และ resolvePlace() ที่ทำการค้นหา + Fuzzy Match กับข้อมูลที่มีอยู่ ก่อนจะตัดสินใจว่าจะใช้ Record เดิมหรือสร้างใหม่

3.3 Daily Operations (Group 2)

กลุ่มนี้รับผิดชอบงานประจำวันที่ต้องทำซ้ำทุกวัน โดยเฉพาะการดึงข้อมูลจาก SCG API และการจับคู่พิกัด 18_ServiceSCG.gs เป็น Commander ของกลุ่ม ทำหน้าที่ดึงข้อมูลจาก SCG API ด้วย Cookie Authentication, แปลง JSON เป็น Flat Rows, คำนวณสรุปตามร้านค้า, เรียก Module 17 จับคู่พิกัด, และสร้างสรุปเจ้าของสินค้า/Shipment ใน V5.4.002 ได้ลบการเรียก populateAliasFromSCGRawData_() ออกจาก Pipeline ของ Group 2 เพื่อปฏิบัติตาม Single Writer Pattern

3.4 Hybrid Alias Architecture

นวัตกรรมสำคัญของ V5.4 คือระบบ Hybrid Alias ที่ผสานข้อดีของ Entity-Specific Alias (V5.2) เข้ากับ Global Alias Ledger ตารางกลาง M_ALIAS ทำหน้าที่เป็น Single Source of Truth สำหรับการค้นหา Alias ข้ามเอนทิตี้ โดยมี master_uuid เป็นกุญแจเชื่อมโยง สถาปัตยกรรมนี้แก้ปัญหาหลัก 3 ประการ: ประการแรกคือ Alias ที่อยู่คนละ Silo ไม่สามารถค้นหาข้ามเอนทิตี้ได้ใน V5.2, ประการที่สองคือ O(N) Scan ใน V4.0 ที่ช้ามากเมื่อข้อมูลเพิ่มขึ้น, และประการที่สามคือไม่มี Fast Track สำหรับการค้นหา ShipToName ที่ใช้บ่อยในงานประจำวัน

Single Writer Pattern: ฟังก์ชัน autoEnrichAliasesFromFactBatch_() ใน 10_MatchEngine.gs เป็นจุดเขียนเดียวสำหรับ M_ALIAS ใน Auto Pipeline ฟังก์ชันอื่นๆ สามารถอ่าน M_ALIAS ได้ แต่ห้ามเขียนโดยเด็ดขาดในระหว่าง Pipeline ทำงาน เพื่อป้องกัน Race Condition และ Data Inconsistency การบังคับใช้ Single Writer นี้ทำให้สามารถรับประกันความสม่ำเสมอของข้อมูล Alias ได้ แม้จะมีหลายกระบวนการทำงานพร้อมกัน

3.5 Match Engine (8 Rules)

Match Engine เป็นหัวใจหลักของ Pipeline ใช้กฎลำดับชั้น (Priority Rules) จำนวน 8 ข้อในการตัดสินใจ โดยประเมินจากกรณีที่แย่ที่สุดไปหาดีที่สุด กฎแต่ละข้อมี Priority Level กำหนดความเร่งด่วน และ Confidence Score บอกระดับความมั่นใจในการตัดสินใจ การออกแบบแบบลำดับชั้นช่วยให้กรณีที่มีปัญหาร้ายแรง (เช่น พิกัดไม่ถูกต้อง) ถูกจัดคิวรีวิวก่อนกรณีที่มีความกำกวมน้อยกว่า

ลำดับกฎActionPriorityเงื่อนไข
1INVALID_LATLNGREVIEW1ไม่มีพิกัดใน Source (0,0 หรือว่าง)
2LOW_QUALITY_DATAREVIEW2ชื่อสั้นเกินไปหรือคุณภาพต่ำ
3GEO_PROVINCE_CONFLICTREVIEW2จังหวัดจากพิกัดไม่ตรงกับจังหวัดในที่อยู่
3.5NEARBY_PENDINGREVIEW1พิกัดใกล้เคียงกัน ต้องมีคนตัดสินใจ
4FULL_MATCHAUTO_MATCH0พบครบทั้ง Person + Place + Geo
5GEO_ANCHORAUTO_MATCH0พบ Geo + อย่างน้อย 1 อย่าง (Person หรือ Place)
6FUZZY_MATCHREVIEW2มีความกำกวม ต้องมีคนตรวจสอบ
7ALL_NEW_WITH_GEOCREATE_NEW0ข้อมูลใหม่ทั้งหมดแต่มีพิกัดถูกต้อง
8DEFAULTREVIEW3กรณีอื่นๆ ที่ไม่เข้าเงื่อนไขใดๆ

กฎที่ 4 (FULL_MATCH) ใช้ Weight รวม: Geo 50% + Person 30% + Place 20% สะท้อนว่าพิกัดมีน้ำหนักสูงสุดเพราะเป็นตัวบ่งชี้ที่เชื่อถือได้มากที่สุดในบริบทขนส่ง ส่วนกฎที่ 5 (GEO_ANCHOR) ปรับน้ำหนักเป็น Geo 60% + Person/Place 40% เพราะเมื่อมีเพียงบางส่วนที่จับคู่ได้ พิกัดจะมีบทบาทสำคัญยิ่งขึ้นในการยืนยันว่าเป็นปลายทางเดียวกัน

4. การประเมินคุณภาพโค้ด

4.2 ปัญหาที่พบ

Critical Issues

C1: 21/26 Entry Points ขาด try-catch
ฟังก์ชันที่ถูกเรียกจาก Menu/Trigger จำนวน 21 จาก 26 ฟังก์ชัน ไม่มีการห่อด้วย try-catch ทำให้เมื่อเกิด Error ระบบจะหยุดทำงานทันทีโดยไม่บันทึก Log และไม่แจ้งเตือนผู้ใช้อย่างเหมาะสม สิ่งนี้ขัดต่อกฎที่ 12 (Error Handling per Entry Point) และอาจทำให้ข้อมูลอยู่ในสถานะกลาง (Inconsistent State) หาก Error เกิดระหว่าง Batch Write
C2: autoEnrichAliasesFromFactBatch_ ยาว 266 บรรทัด

5.1 กลไก Cache 3 ชั้น

ระบบใช้กลไก Cache 3 ชั้นเพื่อลดการอ่าน Google Sheets ที่ช้า: ชั้นที่ 1 เป็น RAM Cache ใช้ Global Variable ใน Script เก็บข้อมูลที่ใช้บ่อยในหน่วยความจำ ชั้นที่ 2 เป็น CacheService ของ GAS เก็บข้อมูลแบบ Serialized JSON ด้วย TTL 6 ชั่วโมง และมีการ Chunk สำหรับข้อมูลที่ใหญ่เกิน 100KB ชั้นที่ 3 เป็น Sheet (Persistent) อ่านจาก Google Sheets โดยตรงเมื่อ Cache ไม่มีข้อมูล การออกแบบ 3 ชั้นนี้ช่วยลดเวลาตอบสนองจากหลายวินาที (อ่าน Sheet) เหลือมิลลิวินาที (RAM Cache) สำหรับข้อมูลที่ใช้ซ้ำ

ชั้นตัวเก็บความเร็วข้อจำกัดใช้สำหรับ
1 - RAMGlobal Variables~0msรีเซ็ตเมื่อ Script รีสตาร์ทGeo Dict, Geo Points
2 - CacheServiceGAS Cache~5-50msTTL 6hr, ขนาดสูงสุด 100KB/keyM_ALIAS, Person, Place
3 - SheetGoogle Sheets~100-500msช้าที่สุด แต่ถาวรข้อมูลทั้งหมด
